I/O

与其它大多数语言一样，haskell也有相应的I/O.haskell 严格地把纯代码与有可能产生副作用的交互代码分隔。

经典I/O

--file : basicio.hs
main = do
    putStrLn "Greetings! What is your name?"
    inpStr <- getLine
    putStrLn $ "Welcome to Haskell," ++ inpStr ++ "!"

有两种方法调用main——runghc basicio.hs 或 在ghci调用main.

程序首先输出putStrLn一个字符串（带有一个换行符）,getLine从标准输入读取一行.<-绑定一个I/O动作的结果到变量。最后程序输入了一段包含输入的文字。

ghci>:type putStrLn
putStrLn :: String -> IO()
ghci>:type getLine
getLine :: IO String

含有IO的类型都包含了一个动作，IO ..表示一个IO动作，并且当动作完成后，会得到一个String,当然这需要运算符<-的帮助。

提示: 也许你看到了一个新的符号$,我们可以通过ghci查看这个符号的作用。

ghci>:info ($)
($) :: (a -> b) -> a -> b       -- Defined in ‘GHC.Base’
infixr 0 $

$接收一个函数和一个参数，并把参数赋给函数，如果我们不使用这个符号，必须把后面的字符串拼接部分用括号括起来。

main在haskell中有特殊意义，main是程序开始执行的地方，其本身就是一个IO动作，类型是IO (),即不返回任何值。这提供了一种隔离机制以实现非纯代码和纯代码之间的隔离。在IO动作中执行IO，并调用纯代码。

do用来定义一串动作（如果是一个动作当然不需要），do代码的值是最后一个动作执行的结果。

下面将上面的代码做了修改，实现了将纯代码和非纯代码隔离:

--file: isolation.hs
greet :: String -> String
greet name =  "Welcome," ++ name ++ "!"

main = do
    putStrLn "Greetings! What is your name?"
    inpStr <- getLine
    putStrLn $ greet inpStr
Pure Impure
相同的输入产生相同的结果 相同的输入可能导致不同的结果
从不会有副作用 可能有副作用
从不修改状态 可能修改程序，系统或者全局状态

使用文件和句柄

Haskell的I/O提供了一些基本函数，这些函数与其他语言中的文件操作类似。

文件读写

-- file:toupper-imp.hs
import System.IO
import Data.Char(toUpper)

main :: IO ()
main = do 
       inh <- openFile "input.txt" ReadMode
       outh <- openFile "output.txt" WriteMode
       mainloop inh outh
       hClose inh
       hClose outh

mainloop :: Handle -> Handle -> IO ()
mainloop inh outh = 
    do ineof <- hIsEOF inh
       if ineof
           then return ()
           else do inpStr <- hGetLine inh
                   hPutStrLn outh (map toUpper inpStr)
                   mainloop inh outh

整个程序的作用是将input.txt文件中的文字全部转化为大写，并保存在output.txt.

首先来看main函数，inh <- openFilee "input.txt" ReadMode打开了文件input.txt并将文件句柄返回给inh.类似地，outh <- openFile "output.txt" WriteMode打开了文件output.txt并将文件句柄返回给outh.
mainloop inh outh对文件进行了操作.hClose inh和hClose outh关闭了文件句柄。

下面看mainloop函数做了什么，mainloop接收了两个文件句柄。
并执行了IO操作，该IO操作循环读取ineof对应文件(input.txt)中的内容，将内容转换为大写并写入outh对应的文件(output.txt)，直到文件读取结束（EOF）.

mainloop中的return含义和常规语言中的return不同，在一般的语言中return用来立即退出当前函数的执行，并返回一个值；haskell中的return执行与<-相反的操作——接收一个纯值，并包装进IO。

关于示例代码中提到的ReadMode和WriteMode说明:这两个值属于IOMode类型

ghci>:info ReadMode
data IOMode = ReadMode | ...    -- Defined in ‘GHC.IO.IOMode’
ghci>:info IOMode
data IOMode = ReadMode | WriteMode | AppendMode | ReadWriteMode
        -- Defined in ‘GHC.IO.IOMode’
instance Eq IOMode -- Defined in ‘GHC.IO.IOMode’
instance Ord IOMode -- Defined in ‘GHC.IO.IOMode’
instance Show IOMode -- Defined in ‘GHC.IO.IOMode’
instance Read IOMode -- Defined in ‘GHC.IO.IOMode’
instance Enum IOMode -- Defined in ‘GHC.IO.IOMode’

可以看到IOMode是代数数据类型.各值含义如下:

IOMode 可读 可写 开始位置 备注
ReadMode 是 否 文件开头 文件必须存在
WriteMode 否 是 文件开头 如果存在,文件会被截断(完全清空)
ReadWriteMode 是 是 文件开头 如果不存在新建文件,如果存在不会损害原来的数据
AppendMode 否 是 文件结尾 如果不存在会新建文件,如果存在不会损害原来的数据

注意:以上操作的都是文本文件,如果需要操作二进制文件,需要使用openBinaryFile替代openFile.

Seek 和 Tell

当从一个对应硬盘上某个文件句柄上读写的时候,操作系统维护了一个当前硬盘位置的内部记录,每次做另一次读的时候,操作系统返回下一个从当前为位置开始的数据块,并且增加这个位置,反映出你正在读的数据.

熟悉python文件独写的读者可能会对seek和tell方法比较熟悉,它们用来改变和获得文件指针位置.haskell中也有类似的函数.

hTell可以找出文件中的指针位置,而hSeek可以改变文件指针位置.

ghci>:m System.IO
ghci System.IO>:info hTell
hTell :: Handle -> IO Integer   -- Defined in ‘GHC.IO.Handle’
ghci System.IO>:info hSeek
hSeek :: Handle -> SeekMode -> Integer -> IO ()
        -- Defined in ‘GHC.IO.Handle’
ghci>:info SeekMode
data SeekMode = AbsoluteSeek | RelativeSeek | SeekFromEnd
        -- Defined in ‘GHC.IO.Device’
instance Eq SeekMode -- Defined in ‘GHC.IO.Device’
instance Ord SeekMode -- Defined in ‘GHC.IO.Device’
instance Show SeekMode -- Defined in ‘GHC.IO.Device’
instance Read SeekMode -- Defined in ‘GHC.IO.Device’
instance Enum SeekMode -- Defined in ‘GHC.IO.Device’

hTell接受一个句柄参数,返回一个整数作为文件位置.hSeek接受一个句柄参数一个SeekMode类型的参数,和一个整数,并返回一个IO动作.其中SeekMode各个值含义如下:

SeekMode 含义
AbsoluteSeek 绝对位置
RelativeSeek 相对位置
SeekFromEnd 尾部位置(距离尾部距离)

标准输入,标准输出,错误

需要了解到,如getLine,putStrLn,print之类的函数,实际上是对句柄的操作.

getLine = hGetLine stdin
putStrLn = hPutStrLn stdout
print = hPrint stdout

其中stdin,stdout,stderr都是句柄类型(Handle)的值.所以可以认为,标准输入输出实际上是一种特殊的文件读写.

使用标准输入输出允许程序和用户进行交互.

例如使用笔记开头的basicio.hs.

$ echo AgileFastx| runghc basicio.hs
Greetings! What is your name?
Welcome,AgileFastx !

当运行basicio.hs时,它不等待键盘的输入,而是从echo程序接受了字符串AgileFastx,结果与使用键盘键入字符串的结果完全一样.

惰性I/O

前面提到的都是传统的I/O例子.由于haskell时一种惰性语言,即除非不要不求值.haskell也提出了一些惰性方法处理I/O.

  • hGetContents

hGetContents函数,接受一个文件句柄,并返回IO String类型,返回的String表示句柄所给文件力的所有数据.

在严格求值的语言中,使用这种方式对于计算机的内存是非常不友好的,因为其一次性将文件内容读入内存,无论文件内容有多少,计算机有可能会因为文件过大而缺少内存存储这些数据,最终导致崩溃.一般情况下,在严格求值语言中,采用循环的方法出处理文件中的全体数据.

haskell中的hGetContents不同,它返回的String是惰性估值的,在调用hGetContents时,实际上没有读取任何内容.当字符串中的字符被处理时,数据才会从句柄读出.只要字符没有被处理,haskell的垃圾回收器会自动释放那块内存.

我们将上面的toupper-imp.hs文件进行适当的修改,以适应这种惰性方法.

--file : toupper-lazy.hs
import System.IO
import Data.Char(toUpper)

main :: IO ()
main = do
    inh <- openFile "input.txt" ReadMode
    outh <- openFile "output.txt" WriteMode
    inpStr <- hGetContents inh
    let result = processData inpStr
    hPutStr outh result
    hClose inh
    hClose outh

processData :: String -> String
processData = map toUpper

修改后,无论处理的文件有多大程序都可以很好地运行.

需要注意的一点是,当使用完成hGetContents后,haskell系统将决定hGetContents返回的字符串被垃圾回收掉,也就是说这个字符串在后续程序中将不再被使用,此时文件将会自动被关闭,因此可以不使用hClose去关闭文件句柄.但是使用hClose仍然是一个很好的习惯.

另一件事是,由于hGetContents是惰性的,在使用hGetContents时,不能随意关闭句柄,直到对完成对结果的使用.这个特性可以概括理解为"非必要不读入",因此如果在必要读入前关闭句柄,这显然会造成麻烦.

--file : toupper-lazy-trouble.hs

import System.IO
import Data.Char(toUpper)
main :: IO ()
main = do 
    inh <- openFile "input.txt" ReadMode
    outh <- openFile "output.txt" WriteMode
    inpStr <- hGetContents inh
    hClose inh
    let result = processData inpStr
    hPutStr outh result
    hClose outh

processData :: String -> String
processData  = map toUpper

程序在显式调用句柄后直接草率地关闭inh句柄,实际上,在后续的语句中,程序仍然需要从句柄中读取内容,这会导致问题.如下:

*** Exception: input.txt: hGetContents: illegal operation (delayed read on closed handle)
  • readFile 和 writeFile

readFile 和 writeFile 集成了一系列文件操作来实现读取文件和写入文件.这两个函数在内部使用了hGetContent.

ghci>:type readFile
readFile :: FilePath -> IO String
ghci>:type writeFile
writeFile :: FilePath -> String -> IO()

再次修改toupper-lazy.hs文件.

--file toupper-lazy-better.hs
import Data.Char(toUpper)

main = do
    inpStr <- readFile "input.txt"
    writeFile "output.txt" (map toUpper inpStr)

不难发现,在使用readFile 和 writeFile时没有暴露句柄,所以就没有必要去使用hClose进行关闭操作,从一定程度上降低了出错的机率.

interact

恰如getLine对应hgetLine等一系列标准输入输出的操作,readFile和writeFile也有一个对应的函数interact.

ghci>:info interact
interact :: (String -> String) -> IO ()         -- Defined in ‘System.IO’

interact接受一个类型为String->String的函数,并将getContents(对应hgetContents的标准输入)的结果传递给这个函数,即惰性读取标准输入,这个函数的结果会发送到标准输出.

例如:

--file: toupper-lazy-interact.hs

import Data.Char(toUpper)

main = interact (map toUpper)

执行后,可以实现将小写字母转换为大写的功能(提示:按Ctrl-C可以结束,或者输入Ctrl-Z结束).当然,我们还可以使用shell的特行,将标准输入输出重定向,实现将input.txt中的内容转化为大写后写入到output.txt.
并显示.

$runghc toupper-lazy-interact.hs < input.txt > output.txt && type output.txt

或者想直接显示转换内容也可以去掉输出重定向.

$runghc toupper-lazy-interact.hs < input.txt

IO Monad

Actions

大多数语言在代码纯度上没有明显的隔离行为,haskell的函数是类似数学意义上的函数——执行一个纯粹的计算,并且这些计算不会被外部所影响,且会返回值.

因此haskell的函数时不能使用I/O的,haskell中使用I/O的工具称为Actions,动作类似于函数,它们在定义的时候不做任何事情,但在被调用时执行任务.I/O动作被定义在IO Monad

串联化

do代码块实际上时把操作链接在一起的快捷记号,有两个运算符可以用来代替do代码块:>>和>>=.

ghci>:type (>>)
(>>) :: Monad m => m a -> m b -> m b
ghci> :type (>>=)
(>>=) :: Monad m => m a -> (a -> m b) -> m b

>>运算符将连个操作串联起来,先运行第一个操作,再运行第二个操作,计算结果是第二个操作的结果,第一个操作的结果被丢弃.

示例如下:

ghci>a <- getLine >> getLine
line1
line2
ghci>a
"line2"

>>=运算符运行一个操作,然后把它的结果传递给一个返回操作的函数.这样地热个操作可以同样运行,而且整个表达式的结果就是第二个操作的结果.

ghci>getLine >>= putStrLn
line1
line1

return

haskell中的return函数于常规语言中的return不同.在haskell中,return用来在Monad里面包赚数据,当使用return的时候,它将一个纯数据带入IO Monad.

ghci>f = \_ -> return "io output"
ghci>:type f
f :: Monad m => p -> m [Char]
ghci>a = f 1
ghci>:type a
a :: Monad m => m [Char]
ghci>b <- a 
ghci>:type b
b :: [Char]

惰性I/O的副作用

haskell中纯代码其实并不完全是无副作用的,但是惰性I/O可以将一些明显触发副作用命令的语句分离开.这些明显触发副作用包括修改全局变量,请求I/O等.

缓冲区

I/O子系统是现代计算机中最慢的部分之一。完成一次写磁盘的时间是一次写内存的几千倍。在网络上的写入还要慢成百上千倍。就算你的操作没有直接和磁盘通信，可能数据被缓存了，I/O还是需要一个系统调用，这个也会减慢速度。

由于这个原因，现代操作系统和编程语言都提供了工具来帮助程序当涉及到I/O的时候更好地运行。操作系统一般采用缓存（Cache），把频繁使用的数据片段保存在内存中，这样就能更快的访问了。

编程语言通常采用缓冲区。就是说，它们可能从操作系统请求一大块数据，就算底层代码是一次一个字节地处理数据的。通过这样，它们可以实现显著的性能提升，因为每次向操作系统的I/O请求带来一次处理开销。缓冲区允许我们去读相同数量的数据可以用少得多的I/O请求。
—— << Real World Haskell >>

缓冲区模式

haskell中有3中不同的缓冲区模式——NoBuffering,LineBuffering和BlockBuffering.

  • NoBuffering没有缓冲区,如hGetLine这样的函数读取的数据是从从左系统一次一个字符读取的.写入的数据会立即写入,同样一次一个字符.NoBuffering通常性能很差.

  • LineBuffering当换行符输出的时候会让输出缓冲区写入,或者当缓冲区太大的时候.这个模式经常是默认模式.

  • BlockBuffering让Haskell在可能的情况下,以一个固定的块大小读取或者写入数据.这在批处理大量数据时性能很好,但对于交互程序是不能使用的,因为它会阻塞输入直到一整块数据被读取.

默认的缓冲区模式依赖于操作系统和haskell的实现.

刷新缓冲区

对于任何类型的缓冲区,可能有时候需要强制haskell写出所有保存在缓冲区里的数据.有时这个会自动发生,如hClose的调用.
